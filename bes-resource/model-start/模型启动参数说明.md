## Docker 运行参数

### 基础容器配置

| 参数 | 说明 | 示例值 |
|------|------|--------|
| `--name` | 容器名称，用于标识和管理容器 | `vllm-qwen3-0.6b` |
| `--rm` | 容器退出后自动删除 | 无参数值 |
| `-it` | 交互式终端模式（用于调试） | 无参数值 |
| `--privileged` | 赋予容器特权模式，访问硬件设备 | 无参数值 |
| `--shm-size` | 共享内存大小 | `16g` |

### 设备挂载

| 参数 | 说明 | 必需性 |
|------|------|--------|
| `--device /dev/davinci[N]` | NPU 设备，N 为设备编号 | 必需 |
| `--device /dev/davinci_manager` | NPU 管理设备 | 必需 |
| `--device /dev/devmm_svm` | 设备内存管理 | 必需 |
| `--device /dev/hisi_hdc` | 华为设备通信 | 必需 |

### 目录挂载

| 参数 | 说明 | 示例 |
|------|------|------|
| `-v <host_path>:<container_path>:ro` | 模型文件目录（只读） | `/home/bes/work/vllm-project/models/Qwen:/models:ro` |
| `-v /usr/local/Ascend/driver:/usr/local/Ascend/driver` | Ascend 驱动目录 | 必需 |
| `-v /usr/local/dcmi:/usr/local/dcmi` | DCMI 工具目录 | 必需 |
| `-v /usr/local/bin/npu-smi:/usr/local/bin/npu-smi` | NPU 系统管理接口 | 必需 |

### 网络配置

| 参数 | 说明 | 示例 |
|------|------|------|
| `-p <host_port>:<container_port>` | 端口映射，主机端口:容器端口 | `-p 9001:8000` |

容器内默认使用 `8000` 端口，映射到主机的不同端口以避免冲突：
- `9000`: Mooncake KV 传输配置
- `9001`: 单机标准配置
- `9002`: 临时调试容器

### 环境变量

| 参数 | 说明 | 示例值 |
|------|------|--------|
| `-e ASCEND_VISIBLE_DEVICES` | 指定可见的 NPU 设备编号 | `4` 或 `7` |
| `-e LD_LIBRARY_PATH` | 动态链接库路径 | `/usr/local/Ascend/driver/lib64:$LD_LIBRARY_PATH` |

## vLLM 服务参数

### 基础服务参数

| 参数 | 说明 | 默认值 | 示例 |
|------|------|--------|------|
| `vllm serve` | 启动 vLLM 服务命令 | - | `vllm serve /models/Qwen3-0.6B` |
| `<model_path>` | 模型路径（第一个位置参数） | - | `/models/Qwen3-0.6B` |
| `--port` | 服务监听端口 | `8000` | `--port 8000` |
| `--host` | 服务监听地址 | `localhost` | `--host 0.0.0.0`（允许外部访问） |
| `--trust-remote-code` | 信任远程代码执行 | `False` | 建议启用以支持自定义模型 |
| `--served-model-name` | 对外暴露的模型名称 | 模型目录名 | `--served-model-name qwen3` |

### 性能与内存参数

| 参数 | 说明 | 推荐值 | 示例 |
|------|------|--------|------|
| `--max-model-len` | 模型最大上下文长度 | 根据模型配置 | `--max-model-len 4096` |
| `--gpu-memory-utilization` | GPU/NPU 内存利用率 | `0.8-0.95` | `--gpu-memory-utilization 0.9` |

**参数说明：**
- `max-model-len`: 控制模型可以处理的最大 token 数量，影响 KV cache 大小
- `gpu-memory-utilization`: 控制模型和 KV cache 占用的内存比例，过高可能导致 OOM

### 高级功能参数

#### KV Transfer Config (Mooncake)

`--kv-transfer-config` 参数用于配置 KV cache 传输，支持分布式推理和 KV cache 复用。

**JSON 配置格式：**
```json
{
  "kv_connector": "MooncakeConnector",
  "kv_role": "kv_both",
  "kv_rank": 0,
  "kv_parallel_size": 1
}
```

**字段说明：**

| 字段 | 说明 | 可选值 |
|------|------|--------|
| `kv_connector` | KV 连接器类型 | `MooncakeConnector` |
| `kv_role` | 节点角色 | `kv_producer`（生产者）、`kv_consumer`（消费者）、`kv_both`（两者兼有） |
| `kv_rank` | 节点排名（分布式推理时） | 从 `0` 开始的整数 |
| `kv_parallel_size` | 并行大小 | 通常为 `1` |

**使用场景：**
- 跨节点 KV cache 传输
- Prefill-Decode 分离架构
- 分布式推理优化

## 启动模式对比

### 模式 1: 交互式调试模式

**脚本：** `start-Qwen3-0.6B-tmp.sh`

**特点：**
- 使用 `-it` 进入交互式 bash 终端
- 不自动启动服务，适合手动调试
- 需要在容器内手动执行启动命令

**适用场景：**
- 调试环境配置问题
- 测试模型加载
- 排查启动错误

**启动命令：**
```bash
docker run -it --rm \
  --name vllm-tmp \
  --privileged \
  --device /dev/davinci4 \
  --device /dev/davinci_manager \
  --device /dev/devmm_svm \
  --device /dev/hisi_hdc \
  -v /home/bes/work/vllm-project/models/Qwen:/models:ro \
  -v /usr/local/Ascend/driver:/usr/local/Ascend/driver \
  -v /usr/local/dcmi:/usr/local/dcmi \
  -v /usr/local/bin/npu-smi:/usr/local/bin/npu-smi \
  -p 9002:8000 \
  --shm-size=16g \
  -e ASCEND_VISIBLE_DEVICES=4 \
  -e LD_LIBRARY_PATH=/usr/local/Ascend/driver/lib64:$LD_LIBRARY_PATH \
  vllm-ascend-0.20.2rc:offline bash
```

---

### 模式 2: 标准单机服务

**脚本：** `start-Qwen3-0.6B-single.sh`

**特点：**
- 直接启动 vLLM 服务
- 容器启动后自动运行服务
- 使用标准内存配置

**适用场景：**
- 生产环境部署
- 标准推理服务
- 独立运行模型

**启动命令：**
```bash
docker run --rm \
  --name vllm-qwen3-0.6b \
  --privileged \
  --device /dev/davinci4 \
  --device /dev/davinci_manager \
  --device /dev/devmm_svm \
  --device /dev/hisi_hdc \
  -v /home/bes/work/vllm-project/models/Qwen:/models:ro \
  -v /usr/local/Ascend/driver:/usr/local/Ascend/driver \
  -v /usr/local/dcmi:/usr/local/dcmi \
  -v /usr/local/bin/npu-smi:/usr/local/bin/npu-smi \
  -p 9001:8000 \
  --shm-size=16g \
  -e ASCEND_VISIBLE_DEVICES=4 \
  -e LD_LIBRARY_PATH=/usr/local/Ascend/driver/lib64:$LD_LIBRARY_PATH \
  vllm-ascend-0.20.2rc:offline \
  vllm serve /models/Qwen3-0.6B \
  --port 8000 \
  --host 0.0.0.0 \
  --trust-remote-code \
  --served-model-name qwen3 \
  --max-model-len 4096 \
  --gpu-memory-utilization 0.9
```

---

### 模式 3: Mooncake KV 传输模式

**脚本：** `start-Qwen3-0.6B-mooncake.sh`

**特点：**
- 启用 Mooncake KV connector
- 支持 KV cache 跨节点传输
- 使用 `bash -c` 执行多行命令

**适用场景：**
- 分布式推理
- Prefill-Decode 分离
- KV cache 复用和传输

**启动命令：**
```bash
docker run --rm \
  --name vllm-qwen3-0.6b-mooncake \
  --privileged \
  --device /dev/davinci7 \
  --device /dev/davinci_manager \
  --device /dev/devmm_svm \
  --device /dev/hisi_hdc \
  -v /home/bes/work/vllm-project/models/Qwen:/models:ro \
  -v /usr/local/Ascend/driver:/usr/local/Ascend/driver \
  -v /usr/local/dcmi:/usr/local/dcmi \
  -v /usr/local/bin/npu-smi:/usr/local/bin/npu-smi \
  -p 9000:8000 \
  --shm-size=16g \
  -e ASCEND_VISIBLE_DEVICES=7 \
  vllm-ascend-0.20.2rc:offline \
  bash -c "
    source /usr/local/Ascend/cann/set_env.sh && \
    export LD_LIBRARY_PATH=/usr/local/lib:\$LD_LIBRARY_PATH && \
    vllm serve /models/Qwen3-0.6B \
    --port 8000 \
    --host 0.0.0.0 \
    --trust-remote-code \
    --served-model-name qwen3 \
    --max-model-len 4096 \
    --gpu-memory-utilization 0.9 \
    --kv-transfer-config '{\"kv_connector\": \"MooncakeConnector\", \"kv_role\": \"kv_both\", \"kv_rank\": 0, \"kv_parallel_size\": 1}'"
```

## 常见使用场景

### 场景 1: 更换模型

**修改项：**
1. 模型路径：`-v /path/to/your/model:/models:ro`
2. vLLM 参数：`vllm serve /models/YourModelName`
3. 容器名称：`--name vllm-your-model`

**示例：**
```bash
# 将 Qwen3-0.6B 替换为 Llama-7B
-v /path/to/llama-7b:/models:ro
vllm serve /models/llama-7b
--name vllm-llama-7b
```

---

### 场景 2: 更换 NPU 设备

**修改项：**
1. 设备编号：`--device /dev/davinci[N]`
2. 环境变量：`-e ASCEND_VISIBLE_DEVICES=N`

**示例：**
```bash
# 使用 NPU 0
--device /dev/davinci0
-e ASCEND_VISIBLE_DEVICES=0

# 使用 NPU 1
--device /dev/davinci1
-e ASCEND_VISIBLE_DEVICES=1
```

**注意：** 设备编号必须与 `ASCEND_VISIBLE_DEVICES` 一致。

---

### 场景 3: 调整内存配置

**根据模型大小调整：**

| 模型参数量 | 推荐 max-model-len | 推荐 gpu-memory-utilization |
|-----------|-------------------|---------------------------|
| 0.6B - 1.5B | 4096 - 8192 | 0.85 - 0.95 |
| 7B - 13B | 2048 - 4096 | 0.80 - 0.90 |
| 30B - 70B | 1024 - 2048 | 0.75 - 0.85 |

**示例：**
```bash
# 小模型 - 高内存利用率
--max-model-len 8192 \
--gpu-memory-utilization 0.95

# 大模型 - 降低上下文长度
--max-model-len 2048 \
--gpu-memory-utilization 0.80
```

---

### 场景 4: 多实例部署

**关键点：**
1. 使用不同的容器名称
2. 使用不同的主机端口
3. 使用不同的 NPU 设备

**示例：**
```bash
# 实例 1 - NPU 4, 端口 9001
--name vllm-qwen-instance1 \
--device /dev/davinci4 \
-e ASCEND_VISIBLE_DEVICES=4 \
-p 9001:8000

# 实例 2 - NPU 5, 端口 9002
--name vllm-qwen-instance2 \
--device /dev/davinci5 \
-e ASCEND_VISIBLE_DEVICES=5 \
-p 9002:8000
```

## 常见问题排查

### 问题 1: 端口冲突

**错误信息：** `bind: address already in use`

**解决方案：**
```bash
# 查看端口占用
netstat -tlnp | grep 9000

# 更改主机端口映射
-p 9003:8000  # 使用未被占用的端口
```

---

### 问题 2: NPU 设备不可用

**错误信息：** `failed to open device` 或 `device not found`

**排查步骤：**
```bash
# 1. 检查 NPU 设备
npu-smi info

# 2. 检查设备文件
ls -la /dev/davinci*

# 3. 确认驱动加载
lsmod | grep dvpp
```

**解决方案：**
- 确保 NPU 驱动已正确安装
- 检查设备编号是否正确
- 确认用户有设备访问权限

---

### 问题 3: 模型加载失败

**错误信息：** `model not found` 或 `failed to load model`

**排查步骤：**
```bash
# 1. 检查模型路径
ls -la /home/bes/work/vllm-project/models/Qwen/

# 2. 检查容器内挂载
docker exec -it vllm-qwen3-0.6b ls /models/

# 3. 验证模型配置文件
cat /path/to/model/config.json
```

**解决方案：**
- 确认主机路径正确
- 确保模型文件完整
- 检查模型格式兼容性

---

### 问题 4: 内存不足 (OOM)

**错误信息：** `CUDA out of memory` 或 NPU OOM

**解决方案：**
```bash
# 降低内存利用率
--gpu-memory-utilization 0.80

# 减小上下文长度
--max-model-len 2048

# 限制最大并发请求数（可选）
--max-num-seqs 32
```

## 性能优化建议

### 1. 内存优化

- **小模型（< 2B）**: `--gpu-memory-utilization 0.95`, `--max-model-len 8192`
- **中等模型（7B-13B）**: `--gpu-memory-utilization 0.90`, `--max-model-len 4096`
- **大模型（30B+）**: `--gpu-memory-utilization 0.85`, `--max-model-len 2048`

### 2. 推理速度优化

- 启用 tensor parallelism（多 NPU 并行）
- 调整 batch size 参数
- 使用量化模型减少内存占用

### 3. 网络优化

- 使用 `--host 0.0.0.0` 允许外部访问
- 根据网络环境调整超时参数
- 使用负载均衡器管理多实例
